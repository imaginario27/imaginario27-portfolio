# Guía de Portfolio

**UI/UX Designer & Frontend Developer · Nuxt · TypeScript · AI-driven workflow**

---

## 1. Principios estratégicos

**El diferencial antes que el catálogo**
El portfolio no es un listado de proyectos. Es una demostración de criterio. Cada decisión de diseño o arquitectura debe estar explicada y justificada.

**Perfil híbrido como ventaja**
No fragmentar la identidad en perfiles separados. Una sola voz, un solo perfil que diseña y construye. Los proyectos muestran la profundidad en cada área.

**Criterio sobre herramientas**
Cualquiera puede generar código con IA. No cualquiera puede hacerlo con sentido arquitectónico y criterio técnico. Ese es el argumento central.

**Menos proyectos, más completos**
Mejor 4 proyectos bien documentados con proceso y decisiones que 15 ejercicios sin contexto. Calidad sobre cantidad.

**Valor para el cliente/empresa**
Cada sección debe responder a la pregunta: ¿qué gana una empresa contratando este perfil? No hablar de habilidades en abstracto sino de impacto concreto.

## 2. El hero: comunicar valor en segundos

El hero debe funcionar sin animación. La frase base tiene que convencer sola:

> _"Diseño interfaces y las construyo yo mismo."_

El texto dinámico animado rota entre los conceptos clave del perfil:

- UI/UX Designer
- Frontend Developer
- AI-driven workflow

**CTAs principales:** Ver proyectos · Contactar

> ⚠️ El diseño del hero es en sí mismo una demostración de tu nivel. Cuídalo especialmente.

## 3. Estrategia de proyectos

### Categorías públicas recomendadas

**Trabajo profesional**
Proyectos del trabajo actual (anonimizados si es necesario). Prioridad máxima. Sistema de diseño, apps Vue/Nuxt/TypeScript, integraciones IA.

**UI/UX**
Casos de estudio con proceso completo. Incluir rol individual si fue trabajo en equipo.

**Frontend**
Proyectos técnicos propios con demo en vivo y enlace a GitHub. Solo los que usen el stack actual.

**Exploración / IA**
Proyectos experimentales con tecnologías emergentes. Incluir contexto: qué explorabas y qué aprendiste. GPT-4 Vision App encaja aquí.

### Qué debe tener cada proyecto

- Contexto: qué problema resolvía y para quién
- Decisiones tomadas: por qué este stack, este enfoque, esta arquitectura
- Proceso: etapas, herramientas, iteraciones
- Resultado: qué funcionó, qué cambiarías
- Demo en vivo (obligatorio en proyectos frontend)
- Enlace a GitHub con código limpio y README
- Mockups o capturas de calidad — no pantallazos pequeños

### Proyectos a ocultar o marcar como privados

- Ejercicios de curso reconocibles: Quizzical, Tenzies, Rick and Morty API
- Proyectos sin TypeScript ni stack actual
- Proyectos sin contexto ni documentación

> 💡 Implementación técnica: añadir campo `visible: boolean` en WordPress y filtrar en la query GraphQL.

## 6. Aspectos técnicos del nuevo portfolio

| Aspecto         | Detalle                                                                                                                    |
| --------------- | -------------------------------------------------------------------------------------------------------------------------- |
| Stack           | Nuxt 3 + TypeScript + sistema de diseño propio + GraphQL conectado a WordPress como headless CMS.                          |
| Campo visible   | Añadir campo `visible: boolean` en WordPress para filtrar proyectos públicos/privados desde la query GraphQL.              |
| Idiomas         | Tres versiones: español, inglés y alemán. El alemán es un diferencial real para el mercado DACH.                           |
| Performance     | Core Web Vitals en verde. El portfolio es en sí mismo una demostración de tu criterio técnico.                             |
| Accesibilidad   | WCAG AA como mínimo. Coherente con lo que declaras en el perfil.                                                           |
| GitHub          | Proyectos activos y mantenidos en la página principal. Proyectos de curso o sin actividad ocultados.                       |
| Demo en vivo    | Obligatorio en proyectos frontend públicos. Sin demo, el proyecto pierde la mitad de su valor.                             |
| Blog (opcional) | Solo si se puede mantener con consistencia. Al menos 1 artículo al mes. Sin actividad reciente, quitar del menú principal. |

## 7. Proyectos de empresa: cómo mostrarlos sin exponer código

**Casos de estudio sin código**
Documentar el problema, las decisiones tomadas, el proceso y el resultado. Sin mostrar código propietario. El valor está en el criterio, no en el código.

**Capturas y mockups anonimizados**
Mostrar interfaces con datos ficticios o difuminados si hay información sensible.

**Métricas sin contexto comprometedor**
"Reduje el tiempo de desarrollo de componentes en un X%" o "el sistema de diseño es usado por N desarrolladores" son datos que puedes mencionar sin revelar nada confidencial.

**AirUI como caso especial**
Es un proyecto personal usado en producción, ya público en GitHub. Es el mejor ejemplo de trabajo real que puedes mostrar sin restricciones.

## 8. Checklist antes de publicar

### Hero

- [ ] La frase principal funciona sin animación
- [ ] El texto dinámico rota entre UI/UX, Frontend e IA
- [ ] Hay dos CTAs visibles: Ver proyectos y Contactar

### Proyectos

- [ ] Al menos 1 proyecto del trabajo actual (aunque sea anonimizado)
- [ ] Cada proyecto tiene contexto, decisiones y resultado
- [ ] Los proyectos frontend tienen demo en vivo
- [ ] Los proyectos de curso están ocultos o en categoría privada
- [ ] React no aparece como skill principal

### Perfil y skills

- [ ] El titular del perfil coincide con el de LinkedIn
- [ ] Los skills están organizados en tres bloques: UI/UX, Frontend, IA
- [ ] No hay listas de iconos sin contexto
- [ ] El CV descargable está actualizado en español e inglés

### Técnico

- [ ] Core Web Vitals en verde
- [ ] Accesibilidad WCAG AA
- [ ] Las tres versiones de idioma están completas
- [ ] El campo `visible` filtra correctamente en GraphQL
- [ ] GitHub muestra proyectos activos y mantenidos
